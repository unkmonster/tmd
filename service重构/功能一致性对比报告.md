# 功能一致性对比报告

## 概述

本报告对比了重构后的代码（tmd）与稳定版代码（tmd-stable）的功能一致性。

---

## 1. Dumper 功能对比

### 稳定版 (tmd-stable/internal/cli/executor.go 和 executor_download.go)

**初始化逻辑：**
```go
// executor.go:47-48
dumper := downloading.NewDumper()
_ = dumper.Load(pathHelper.ErrorJ)
```

**保存逻辑：**
```go
// executor.go:57-62
defer func() {
    if dumper.Count() > 0 {
        dumper.Dump(pathHelper.ErrorJ)
        log.Infof("%d tweets have been dumped", dumper.Count())
    }
}()
```

**失败推文收集逻辑：**
```go
// executor_download.go:25-33
for _, f := range failed {
    eid, err := f.Entity.Id()
    if err != nil {
        log.Warnln("failed to get entity id:", err)
        continue
    }
    dumper.Push(eid, f.Tweet)
}
```

### Refactored 版 (tmd/internal/service/download_service.go)

**初始化逻辑：**
```go
// download_service.go:44-47
dumper := downloading.NewDumper()
if err := dumper.Load(pathHelper.ErrorJ); err != nil {
    log.Warnf("Failed to load dumper: %v", err)
}
defer s.saveDumper(dumper, pathHelper.ErrorJ)
```

**保存逻辑 (saveDumper 方法)：**
```go
// download_service.go:438-442
func (s *downloadServiceImpl) saveDumper(dumper *downloading.TweetDumper, path string) {
    if err := dumper.Dump(path); err != nil {
        log.Warnf("Failed to save dumper: %v", err)
    }
}
```

**失败推文收集逻辑 (collectFailedTweets 方法)：**
```go
// download_service.go:445-453
func (s *downloadServiceImpl) collectFailedTweets(dumper *downloading.TweetDumper, failedTweets []*downloading.TweetInEntity) {
    for _, tweet := range failedTweets {
        if tweet.Entity != nil {
            if id, err := tweet.Entity.Id(); err == nil {
                dumper.Push(id, tweet.Tweet)
            }
        }
    }
}
```

### 差异分析

| 项目 | 稳定版 | Refactored 版 | 是否一致 |
|------|--------|---------------|----------|
| Dumper 初始化 | `_ = dumper.Load(pathHelper.ErrorJ)` (忽略错误) | `if err := dumper.Load(...); err != nil { log.Warnf(...) }` (记录警告) | **不一致** - Refactored 版增加了错误日志 |
| Dumper 保存条件 | `if dumper.Count() > 0` 才保存 | 无条件保存 | **不一致** - 稳定版有优化，空 Dumper 不写入 |
| 保存错误处理 | 无错误处理 | `log.Warnf("Failed to save dumper: %v", err)` | **不一致** - Refactored 版增加了错误日志 |
| 失败推文收集 | 检查 `err != nil`，失败时 `log.Warnln` 并 `continue` | 检查 `tweet.Entity != nil` 和 `err == nil` | **不一致** - 错误处理逻辑相反 |
| 空 Entity 处理 | 获取 ID 失败时记录警告并跳过 | 直接跳过，无日志 | **不一致** - 稳定版有警告日志 |

**关键差异：**
1. **稳定版在 `dumper.Count() > 0` 时才保存**，Refactored 版无条件保存
2. **稳定版在获取 Entity ID 失败时会记录警告日志**，Refactored 版静默跳过
3. **Refactored 版在加载 Dumper 失败时记录警告**，稳定版静默忽略

## 3. Profile 下载功能对比

### 稳定版 (executor_profile.go)

**入口函数：**
```go
// handleProfileDownload:36-143
func handleProfileDownload(ctx context.Context, cfg *CLIConfig, deps *Dependencies, task *Task, pathHelper *StorePath, dwn downloader.Downloader, fileWriter downloader.FileWriter, versionManager downloader.VersionManager) {
    clients := make([]*resty.Client, 0)
    clients = append(clients, deps.Client)
    clients = append(clients, deps.AdditionalClients...)

    storage, err := profile.NewFileStorageManager(pathHelper.Users)
    if err != nil {
        log.Fatalln("failed to create profile storage:", err)  // Fatal!
    }
    storage.SetVersionManager(versionManager)

    profileDownloader := profile.NewProfileDownloaderWithDB(nil, storage, clients, deps.DB, dwn, fileWriter)
    // ... 构建 requests ...
    results := profileDownloader.DownloadMultiple(ctx, uniqueRequests)
    
    // 统计结果
    success := 0
    failed := 0
    skipped := 0
    for _, r := range results {
        if r.Success {
            success++
        } else if r.Error != nil {
            failed++
        } else {
            skipped++
        }
    }
    log.Infoln("profile download completed - total:", len(results), "success:", success, "failed:", failed, "skipped:", skipped)
    
    // 打印结果表格
    fmt.Println("\n=== PROFILE_DOWNLOAD_RESULTS ===")
    // ...
}
```

**关键特性：**
1. 使用 `profile.NewProfileDownloaderWithDB` (传入 DB)
2. 构建详细的 `DownloadRequest` 包含用户所有信息
3. 去重逻辑：`seen := make(map[string]bool)`
4. 统计成功/失败/跳过三种状态
5. 打印详细结果表格
6. 从列表获取成员时调用 `database.MarkListMembersAccessibleByIDs`

### Refactored 版 (download_service.go:456-500)

```go
func (s *downloadServiceImpl) downloadProfile(ctx context.Context, screenNames []string, pathHelper *path.StorePath, versionManager *downloader.DefaultVersionManager, fileWriter *downloader.DefaultFileWriter, dwn *downloader.DefaultDownloader) {
    if len(screenNames) == 0 {
        return
    }

    log.Infof("Downloading profiles for %d users", len(screenNames))

    storage, err := profile.NewFileStorageManager(pathHelper.Users)
    if err != nil {
        log.Errorf("Failed to create file storage manager: %v", err)
        return  // 不是 Fatal
    }
    storage.SetVersionManager(versionManager)

    pd := profile.NewProfileDownloaderWithClients(
        profile.DefaultConfig(),
        storage,
        append([]*resty.Client{s.deps.Client}, s.deps.AdditionalClients...),
        dwn,
        fileWriter,
    )

    requests := make([]profile.DownloadRequest, len(screenNames))
    for i, screenName := range screenNames {
        requests[i] = profile.DownloadRequest{ScreenName: screenName}  // 只有 ScreenName
    }

    results := pd.DownloadMultiple(ctx, requests)

    var successCount, failCount int
    for _, result := range results {
        if result.Success {
            successCount++
        } else {
            failCount++
        }
    }

    log.Infof("Profile download completed: %d success, %d failed", successCount, failCount)
}
```

### 差异分析

| 项目 | 稳定版 | Refactored 版 | 是否一致 |
|------|--------|---------------|----------|
| Profile Downloader 创建 | `NewProfileDownloaderWithDB(nil, storage, clients, deps.DB, dwn, fileWriter)` | `NewProfileDownloaderWithClients(config, storage, clients, dwn, fileWriter)` | **不一致** - 稳定版使用 DB 版本 |
| 存储创建失败处理 | `log.Fatalln` (程序退出) | `log.Errorf` + `return` (继续执行) | **不一致** - 稳定版更严格 |
| Request 构建 | 包含完整用户信息 (ScreenName, UserTitle, Name, UserID, AvatarURL, BannerURL, Description, Location, URL, Verified, Protected, CreatedAt) | 仅包含 ScreenName | **不一致** - 稳定版信息更完整 |
| 去重逻辑 | 有 `seen` map 去重 | 无去重逻辑 | **不一致** - Refactored 版可能重复下载 |
| 结果统计 | 三种状态：success, failed, skipped | 两种状态：success, failed | **不一致** |
| 结果输出 | 打印详细结果表格 | 仅日志输出统计 | **不一致** - 稳定版输出更详细 |
| 列表成员标记 | `database.MarkListMembersAccessibleByIDs(db, uids)` | 无此调用 | **不一致** - Refactored 版缺少此功能 |

**关键差异：**
1. **稳定版使用 `NewProfileDownloaderWithDB`**，可以利用数据库缓存用户信息
2. **稳定版构建的 Request 包含完整用户信息**，Refactored 版只有 ScreenName
3. **稳定版有去重逻辑**，Refactored 版无去重
4. **稳定版区分 skipped 状态**，Refactored 版不区分
5. **稳定版在存储创建失败时 Fatal**，Refactored 版仅记录错误并返回

## 4. 进度报告对比

### 稳定版

稳定版使用直接的 `log.Infoln`, `log.Errorf` 等日志输出：

```go
log.Infoln("start working for...")
PrintTask(task)
log.Infof("%d tweets have been dumped", dumper.Count())
log.Infoln("profile download completed - total:", len(results), "success:", success, "failed:", failed, "skipped:", skipped)
fmt.Println("\n=== PROFILE_DOWNLOAD_RESULTS ===")
```

### Refactored 版

Refactored 版使用 `ProgressReporter` 接口：

```go
// progress.go:21-25
type ProgressReporter interface {
    OnProgress(taskID string, p Progress)
    OnComplete(taskID string, r Result)
    OnError(taskID string, err error)
}

// 使用示例
reporter.OnProgress(taskID, Progress{Stage: "downloading", Current: screenName})
reporter.OnError(taskID, err)
reporter.OnComplete(taskID, Result{Message: "User download completed"})
```

### 差异分析

| 项目 | 稳定版 | Refactored 版 | 是否一致 |
|------|--------|---------------|----------|
| 报告方式 | 直接日志输出 | 通过 ProgressReporter 接口 | **不一致** - 架构不同 |
| 任务信息 | `PrintTask(task)` 打印用户和列表 | 通过 Progress.Current 传递 | **不一致** |
| 完成统计 | 直接打印结果 | 通过 `OnComplete` 回调 | **不一致** |
| 灵活性 | 固定日志输出 | 可插拔的报告器 (NopReporter, LogReporter) | Refactored 版更灵活 |

**关键差异：**
1. **Refactored 版使用接口抽象**，支持不同的报告实现（CLI 用 LogReporter，GUI 可用其他实现）
2. **稳定版直接输出到日志**，更简单直接
3. **Refactored 版的 LogReporter 输出格式与稳定版不同**

## 5. 错误处理对比

### 稳定版

```go
// 用户获取失败
usr, uid, err := twitter.GetUserByScreenName(ctx, client, screenName)
if err != nil {
    database.MarkUserInaccessible(db, uid, screenName)
    return nil, err
}

// Profile 存储创建失败
storage, err := profile.NewFileStorageManager(pathHelper.Users)
if err != nil {
    log.Fatalln("failed to create profile storage:", err)  // Fatal
}

// 下载错误
failed, err := downloading.BatchDownloadAny(...)
if err != nil {
    return err
}
```

### Refactored 版

```go
// 用户获取失败
user, uid, err := twitter.GetUserByScreenName(ctx, s.deps.Client, screenName)
if err != nil {
    s.userRepo.MarkInaccessible(ctx, uid, screenName)  // 使用 Repository
    reporter.OnError(taskID, err)  // 报告错误
    return err
}

// Profile 存储创建失败
storage, err := profile.NewFileStorageManager(pathHelper.Users)
if err != nil {
    log.Errorf("Failed to create file storage manager: %v", err)
    return  // 非 Fatal，继续执行
}

// 下载错误
failedTweets, err := downloading.BatchDownloadAny(...)
if err != nil {
    reporter.OnError(taskID, err)
    return err
}
```

### 差异分析

| 项目 | 稳定版 | Refactored 版 | 是否一致 |
|------|--------|---------------|----------|
| 用户不可访问标记 | `database.MarkUserInaccessible(db, uid, screenName)` | `s.userRepo.MarkInaccessible(ctx, uid, screenName)` | **一致** (实现不同但功能相同) |
| Profile 存储失败 | `log.Fatalln` (程序退出) | `log.Errorf` + `return` (继续) | **不一致** - 严重程度不同 |
| 错误报告 | 直接返回错误 | `reporter.OnError(taskID, err)` + 返回 | **不一致** - Refactored 版增加回调 |
| Dumper 加载错误 | 静默忽略 (`_ = dumper.Load(...)`) | `log.Warnf("Failed to load dumper: %v", err)` | **不一致** - Refactored 版增加日志 |

## 总结

### 完全一致的功能
- 核心下载流程 (`downloading.BatchDownloadAny` 调用)
- Retry 调用时机和条件
- 基本的 Dumper 使用模式（初始化、收集、保存）

### 存在差异的功能

| 功能模块 | 差异严重程度 | 说明 |
|----------|--------------|------|
| **Dumper 保存条件** | **高** | 稳定版优化了空 Dumper 的保存 |
| **Dumper 错误日志** | 中 | Refactored 版增加了更多日志 |
| **Profile 下载** | **高** | 稳定版功能更完整（去重、详细 Request、skipped 统计） |
| **Profile 存储失败处理** | **高** | 稳定版 Fatal，Refactored 版继续执行 |
| **进度报告** | 中 | 架构不同，Refactored 版使用接口抽象 |
| **结果输出** | 中 | 稳定版输出更详细的表格格式 |
| **错误处理** | 低 | Refactored 版增加了 reporter 回调 |

### 建议修复项

1. **Dumper 保存条件**：Refactored 版应添加 `dumper.Count() > 0` 检查
2. **Profile 下载**：Refactored 版应添加去重逻辑和完整的 Request 构建
3. **Profile 存储失败**：考虑是否应保持 Fatal 行为
4. **MarkListMembersAccessibleByIDs**：Refactored 版的 ListDownload 中缺少此调用
